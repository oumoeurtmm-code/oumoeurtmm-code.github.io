What You're Building

A hardened EC2 instance in a custom VPC — private subnet, no public IP, no open ports. SSM Session Manager replaces SSH entirely.

  • ▸Custom VPC — 10.0.0.0/16 CIDR with public and private subnets in separate AZs
  • ▸Internet Gateway — attached to VPC, routes public subnet outbound traffic
  • ▸NAT Gateway — in public subnet with Elastic IP, gives private instances outbound internet access
  • ▸Security Group — zero inbound rules, egress port 443 only (SSM requirement)
  • ▸EC2 in private subnet — Amazon Linux 2023, t3.micro, no public IP, IAM role for SSM

🧠 Key Concepts Practiced

VPC & Networking

  • ✓What makes a subnet public? — A route to an IGW in its route table. Not a setting on the subnet itself.
  • ✓VPC scope — Spans all AZs in a region. Subnets are locked to a single AZ.
  • ✓IGW vs NAT GW — IGW handles inbound+outbound for public instances. NAT GW handles outbound-only for private instances.
  • ✓AWS Reserved IPs — First 4 and last 1 IP in every subnet are reserved by AWS. A /24 gives you 251 usable IPs.

Security Groups

  • ✓Stateful — Return traffic automatically allowed. If inbound 443 is permitted, responses flow back without an explicit outbound rule.
  • ✓Default behavior — New SGs deny all inbound, allow all outbound. We removed the allow-all egress and replaced with port 443 only.
  • ✓SG vs NACL — SGs are stateful, instance-level. NACLs are stateless, subnet-level. Rules evaluated differently.
  • ✓No deny rules — Security Groups only have allow rules. Everything not explicitly allowed is implicitly denied.

IAM & Identity

  • ✓IAM Roles vs Users — Roles are assumed by services. EC2 uses a role via an instance profile to make API calls without stored credentials.
  • ✓Instance Metadata Service — 169.254.169.254 provides temporary, auto-rotating credentials to the instance at runtime.
  • ✓AmazonSSMManagedInstanceCore — Minimum policy for SSM. Grants ssm:*, ssmmessages:*, and ec2messages:*.
  • ✓Least privilege — No AdministratorAccess on the instance. Only what SSM needs, nothing more.

SSM Session Manager

  • ✓No SSH required — Session Manager connects over HTTPS (443) to SSM endpoints. No inbound ports needed on the instance.
  • ✓IAM-controlled access — Who can connect is governed by IAM policies on the caller, not SSH keys on the instance.
  • ✓CloudTrail logged — Every session start/end is recorded. Optional: log session output to S3 or CloudWatch Logs.
  • ✓Agent registration — SSM agent on the instance reaches out to SSM endpoints via NAT GW. Takes 1-2 minutes after launch.
