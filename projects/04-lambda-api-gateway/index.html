What You're Building

A serverless REST API — the classic entry point to cloud-native development. No EC2, no VPC, no security groups to manage. API Gateway handles HTTP, Lambda runs your code on demand.

  • ▸Lambda function — Python 3.12 handler that responds to GET and POST at /items
  • ▸API Gateway REST API — /items resource with GET and POST methods, LAMBDA_PROXY integration
  • ▸IAM execution role — Lambda assumes this role; grants permission to write CloudWatch Logs
  • ▸CloudWatch Logs — auto-created on first invocation, captures all print() output
  • ▸API stage deployment — "lab" stage forms the public invoke URL

🧠 Key Concepts Practiced

Serverless & Lambda

  • ✓Serverless — No OS, no patching, no capacity planning. AWS manages the execution environment entirely.
  • ✓Event-driven — Lambda only runs when triggered. No invocation = no cost. Perfect for APIs with variable traffic.
  • ✓Cold starts — First invocation (or after idle) spins up a new container: ~100-500ms extra latency. Subsequent calls reuse warm containers.
  • ✓Execution environment reuse — In-memory state (like the ITEMS list) persists between warm invocations on the same container. Never rely on this for stateful data.

API Gateway Patterns

  • ✓LAMBDA_PROXY integration — API Gateway passes the raw HTTP request as a JSON event. Lambda controls the full response (statusCode, headers, body).
  • ✓LAMBDA integration (non-proxy) — API Gateway maps the request using VTL templates before sending to Lambda. More control, more complexity.
  • ✓Stages and deployments — A deployment is a snapshot of your API config. A stage (like "lab") points to a deployment. Changing a method requires a new deployment.
  • ✓Resource-based policy — Lambda permissions use add-permission to allow API Gateway to invoke the function. Scoped by source-arn to the specific API, stage, and method.

IAM for Lambda

  • ✓Execution role — The identity Lambda assumes when running. Controls what AWS resources Lambda can call (DynamoDB, S3, etc.).
  • ✓Resource-based policy — The Lambda function's own policy. Controls who can invoke Lambda (API Gateway, S3 events, other accounts).
  • ✓Two permission models — Execution role = what Lambda can do. Resource policy = who can invoke Lambda. Both must be correct.
  • ✓IAM propagation delay — After creating a role, wait ~10 seconds before using it in Lambda. CloudFormation and scripts must account for this.

Exam Notes

  • ✓Lambda max timeout — 15 minutes (900 seconds). For longer workloads, use Step Functions or ECS Fargate.
  • ✓Lambda layers — Share code/libraries across functions. Up to 5 layers per function. Good for dependencies like pandas, numpy.
  • ✓Concurrency — Default limit: 1,000 concurrent executions per region. Reserved concurrency locks a quota for critical functions.
  • ✓Integration HTTP method — For Lambda integrations, --integration-http-method is always POST, regardless of the API method (GET, POST, DELETE, etc.).
