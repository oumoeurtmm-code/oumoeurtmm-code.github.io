What You're Building

A classic two-tier web application pattern — app tier in a public subnet, data tier in isolated private subnets. The foundation of most production web apps on AWS.

  • ▸3-subnet VPC — 1 public (EC2) + 2 private across 2 AZs (RDS subnet group requirement)
  • ▸EC2 app server — public subnet, public IP, MySQL client, reaches SSM via IGW (no NAT Gateway)
  • ▸RDS MySQL 8.0 — db.t3.micro, private subnets, not publicly accessible
  • ▸SG referencing — RDS SG allows inbound 3306 from EC2 SG object, not a CIDR block
  • ▸Secrets Manager — DB password stored encrypted, retrieved via IAM-controlled API call

🧠 Key Concepts Practiced

RDS & Databases

  • ✓DB Subnet Groups — RDS uses them to pick which subnet to deploy into. Must span 2+ AZs even for single-AZ instances.
  • ✓Publicly Accessible flag — Controls whether RDS gets a public IP. Even with false, it's still reachable within the VPC.
  • ✓Multi-AZ — Synchronous standby replica in a different AZ. Automatic failover if primary fails. Not the same as a read replica.
  • ✓Backup retention = 0 — Disables automated backups. In production, set 7-35 days. We use 0 for faster cleanup in labs.

Security Patterns

  • ✓SG Referencing — RDS SG allows port 3306 from EC2 SG. Rules follow the instance even if its IP changes.
  • ✓Secrets Manager retrieval — EC2 calls get-secret-value at runtime via IAM role. No password in code, config files, or env vars.
  • ✓Defense in depth — RDS is protected by: VPC isolation, private subnets, no public IP, RDS SG (port 3306 only), IAM.
  • ✓No NAT Gateway — EC2 in public subnet with public IP reaches SSM and Secrets Manager directly via IGW.

Architecture Patterns

  • ✓Two-tier architecture — App tier (EC2) separated from data tier (RDS) at the subnet and security group level.
  • ✓Principle of least privilege — EC2 SG only allows what it needs: 443 for SSM, 3306 for RDS. Nothing else.
  • ✓Private data tier — RDS has no public IP. The only path in is through EC2, which itself is controlled by SGs and IAM.
  • ✓User data for bootstrapping — MySQL client installed at launch via EC2 user data script — no manual setup needed.

Exam Notes

  • ✓RDS endpoint — A DNS name that always resolves to the current primary. Doesn't change during Multi-AZ failover.
  • ✓Secrets Manager vs Parameter Store — Secrets Manager: rotation, higher cost ($0.40/secret). Parameter Store SecureString: cheaper, no auto-rotation.
  • ✓gp2 vs gp3 storage — gp3 is newer and cheaper with baseline 3000 IOPS. gp2 scales IOPS with size. Free tier uses gp2.
  • ✓Read replica vs Multi-AZ — Read replica: asynchronous, for read scaling. Multi-AZ: synchronous, for HA/failover only.

Connect to EC2 + RDS

# 1. Connect to EC2 via SSM
aws ssm start-session \
  --target $INSTANCE_ID --region us-east-1

# 2. Get credentials from Secrets Manager
SECRET=$(aws secretsmanager get-secret-value \
  --secret-id aws-cert-study/lab03/db-credentials \
  --query SecretString --output text --region us-east-1)
DB_PASS=$(echo $SECRET | python3 -c \
  "import sys,json; print(json.load(sys.stdin)['password'])")

# 3. Connect to MySQL
mysql -h $RDS_ENDPOINT -u admin -p$DB_PASS labdb
